# arr

A single [orca](https://github.com/scottdkey/orca) plugin for the whole **\*arr
stack** — Sonarr, Radarr, Prowlarr, and Lidarr — in **one cdylib** exposing four
tool namespaces.

## Design: one cdylib, four namespaces

The four \*arr forks share one C# codebase, so their OpenAPI specs are
structurally near-identical and their orca tool surfaces are the same shape
(endpoint CRUD + health/indexers/test_indexers/system_status). Rather than four
separate repos and four cdylibs, this repo ships **one crate** that bundles all
four:

| Namespace | App | API version | Notes |
|---|---|---|---|
| `sonarr.*` | Sonarr | `/api/v3` | TV / `/series` model |
| `radarr.*` | Radarr | `/api/v3` | Movies / `/movie` model |
| `prowlarr.*` | Prowlarr | `/api/v1` | Indexer manager; adds `indexer_status` |
| `lidarr.*` | Lidarr | `/api/v1` | Music / `/artist` model |

Each namespace exposes:

- `<app>.{list, detail, create, update, delete}` — endpoint registry CRUD
  (generated by the toolkit's `endpoint_resource!`).
- `<app>.health` — **detection.** Health issues from `/health`.
- `<app>.indexers` — configured indexers from `/indexer`.
- `<app>.test_indexers` — **remediation.** Re-test every indexer, clearing
  stale backoff.
- `<app>.system_status` — app name / version / instance.
- `prowlarr.indexer_status` — Prowlarr-only per-indexer backoff status.

### Layout

```
src/
  lib.rs          module wiring + crate docs
  abi_export.rs   ONE abi_stable RootModule; manifest()/invoke() filtered to the
                  four owned namespaces (sonarr./radarr./prowlarr./lidarr.)
  clients.rs      INTERNAL shared base library: Flavor/Config/Client transport,
                  result structs, common `ops`, and one progenitor-generated
                  typed-client module per spec (flavor! macro)
  clients/auth.rs crate-internal *arr auth transport composition
  sonarr.rs       sonarr.* tool surface
  radarr.rs       radarr.* tool surface
  prowlarr.rs     prowlarr.* tool surface
  lidarr.rs       lidarr.* tool surface
specs/
  sonarr.openapi.json  radarr.openapi.json  prowlarr.openapi.json
  lidarr.openapi.json  readarr.openapi.json
build.rs          progenitor codegen for every spec under specs/
```

The base library (`clients`) is **internal to this one repo** — it is not a
separately published crate. A fix to the shared transport or ops lands once for
all four apps.

## Adding a flavor

Adding a new \*arr-family app is three small steps:

1. Drop `specs/<flavor>.openapi.json` into the repo. `build.rs` codegens a typed
   client for every spec present.
2. Add `flavor!(<flavor>);` to the list in `src/clients.rs`.
3. Add a sibling `src/<flavor>.rs` tool-surface module, `pub mod <flavor>;` in
   `src/lib.rs`, and `"<flavor>."` to `TOOL_PREFIXES` in `src/abi_export.rs`.

(`specs/readarr.openapi.json` ships a generated client today but no tool surface
— it is wired for a future `readarr.*` namespace.)

## The single orca dependency

The only orca dependency is **`plugin-toolkit`**. Every orca primitive — serde,
schemars, clap, anyhow, thiserror, the dispatch/contract/derive macro targets,
the HTTP transport, endpoint addressing, the tokio runtime, and the whole
progenitor runtime — is reached through `plugin_toolkit::*` and its prelude. The
build-time codegen rewrites the progenitor-generated crate paths to
`::plugin_toolkit::*`, so the generated clients need none of those as direct
deps.

The one genuinely-external dep is **`abi_stable`**: `#[export_root_module]`
expands to bare `::abi_stable` paths at the cdylib FFI boundary orca's
`plugin-loader` `dlopen`s, which cannot route through the toolkit.

## Building

```sh
# With an orca checkout at ../orca, the committed .cargo/config.toml [patch]
# resolves plugin-toolkit to the local in-tree crates:
cargo build --release        # produces target/release/libarr.dylib (the cdylib)
cargo test
cargo clippy --all-targets -- -D warnings
```

Without an `../orca` checkout, `plugin-toolkit` resolves from the pinned git tag
in `Cargo.toml` against the private orca repo.

## Deployment

The \*arr stack runs as Docker containers managed elsewhere in the homelab.
This repo ships the orca **plugin** only — there are no install/update/backup
deploy assets here yet (no Dockerfile/compose/lxc). If lifecycle automation is
wanted later, add a `lifecycle.rs` exposing install/update/backup as
`#[orca_tool]`s, mirroring the jellyfin/plex repos.
